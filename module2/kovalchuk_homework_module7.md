Варіант: **A** — аналіз життєвого циклу контейнера.

Образ: `python:3-slim`, застосунок — вбудований HTTP-сервер `python3 -m http.server 8080`.

---

## 1. Запуск контейнера

```bash
docker run -d --name hw7 -p 8080:8080 python:3-slim python3 -m http.server 8080
```
```
610c5b9189b4d6781589272abd75fafb8a30eb548b2abb475f736e5d20341d50
```

Перевірка статусу:

```bash
docker ps --filter name=hw7
```
```
NAMES   IMAGE           STATUS         PORTS
hw7     python:3-slim   Up 2 seconds   0.0.0.0:8080->8080/tcp, [::]:8080->8080/tcp
```

`-d` — у фоні (detached), `--name hw7` — ім'я контейнера, `-p 8080:8080` — проброс порту хост→контейнер, останній аргумент — команда запуску всередині.

## 2. Процес PID 1 усередині контейнера

Образ `slim` не містить утиліти `ps`, тож процес переглядаємо через псевдофайлову систему `/proc/1`:

```bash
docker exec hw7 sh -c 'cat /proc/1/cmdline | tr "\0" " "; echo; grep -E "^(Name|State|Pid)" /proc/1/status'
```
```
python3 -m http.server 8080
Name:	python3
State:	S (sleeping)
Pid:	1
```

**PID 1 — це `python3 -m http.server 8080`**. Саме цю команду вказано при `docker run`, а всередині контейнера немає системи ініціалізації (systemd/init), тому перший запущений процес отримує номер 1.

PID 1 і визначає життя контейнера: **контейнер працює доти, доки працює його PID 1**. Як тільки PID 1 завершується — Docker зупиняє контейнер.

## 3. Завершення: docker stop

```bash
time docker stop hw7
```
```
hw7

real    0m1.705s
user    0m0.091s
sys     0m0.076s
```

**Який сигнал отримує процес:** `docker stop` спершу посилає процесу з PID 1 сигнал **SIGTERM** — прохання завершитись коректно (дочистити ресурси, закрити з'єднання). `python -m http.server` реагує на SIGTERM і завершується, тому зупинка займає лише ~1.7 с.

**Якщо процес ігнорує SIGTERM:** Docker чекає завершення grace period (за замовчуванням **10 секунд**, регулюється через `docker stop -t` / `--stop-timeout`). Після нього посилається **SIGKILL** — який перехопити або ігнорувати неможливо, ядро примусово вбиває процес. У такому разі зупинка триває рівно стільки, скільки триває grace period.

## 4. Логи

Згенеруємо запит до сервера й подивимось логи:

```bash
curl -s -o /dev/null http://localhost:8080/
docker logs hw7
```
```
172.17.0.1 - - [07/Jul/2026 19:46:00] "GET / HTTP/1.1" 200 -
172.17.0.1 - - [07/Jul/2026 19:46:00] "GET / HTTP/1.1" 200 -
```

**Звідки беруться логи:** Docker перехоплює stdout/stderr процесу PID 1 (і його дочірніх процесів) і зберігає їх через драйвер логів (за замовчуванням `json-file` на диску хоста). `docker logs` просто виводить цей буфер. `python -m http.server` пише кожен запит у stdout — це рядки `"GET / HTTP/1.1" 200`. Після `docker stop` логи не зникають, їх досі видно через `docker logs`, поки контейнер не видалено (`docker rm`).

## Висновок

Життя контейнера визначає **PID 1**. Контейнер завершується, коли PID 1 завершується — добровільно, по SIGTERM після `docker stop`, або примусово по SIGKILL після закінчення grace period.
